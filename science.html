原理科普

AI 团队是怎么工作的？

5 分钟，从零到懂。
不讲模型参数，只讲你关心的事。

第一章

Agent 不是聊天机器人。

人们以为 AI 就是能聊天的程序。错了。


聊天机器人是应答式的：你问，它答，结束。

Agent 是行动式的：你说目标，它拆解任务，调用工具，自主执行，反馈结果。


两者的区别，就是接线员和员工的区别。


Agent 的核心能力：

  • 👁️感知：读文件、搜网络、看数据
  • 🧠思考：拆解任务、制定计划、判断优先级
  • ⚡行动：调 API、写代码、发消息、存数据
  • 📊反馈：报告结果、处理异常、等待下一个指令

Larry 的技术专家⚙️就是这样工作的。你说"把这份报告部署到服务器"，它自己找 GitHub、下载、配置、测试、上线。你不需要教每一步。

"ChatGPT 是个聪明的外包，Agent 是个能独立完成工作的同事。你想要的是哪个？"

— Larry
第二章

技能是 Agent 的超能力模块。

一个空白的 Agent 什么都做不了。就像一个聪明的新员工，没有培训，也交不出活。


Skill 就是给 Agent 的技能培训包。


每个 Skill 包含：

  • 📌这个技能用来干什么（定义）
  • ⚡遇到什么情况触发（触发条件）
  • 🔧用哪些工具来执行（工具调用）
  • ✅怎么算做完了（完成标准）

Larry 的笔杆子✍️有个叫 baoyu-post-to-wechat 的 Skill，触发条件是"文章写完且审核通过"，执行步骤是"格式化→上传封面图→发布→回传链接"，做完自动通知 Larry。

"Skill 是知识的固化。你教了一次，它记住了，以后永远这样干。人类员工会忘，Skill 不会。"

— Larry

你不需要每次都重新教 Agent 怎么做同一件事。封装成 Skill，一劳永逸。

第三章

AI 说假话，不是它坏，是它不知道真相。

语言模型有个根本缺陷：训练截止日期之后的事它不知道，你自己的私有数据它更不知道。


你问它你公司的报销流程，它只能编一个听起来合理的答案。


RAG（检索增强生成）就是解决这个问题的。


工作原理很简单：

  • 1️⃣把你的文档全部向量化，建立索引
  • 2️⃣用户提问时，先检索相关文档片段
  • 3️⃣把检索到的真实内容 + 用户问题一起送给 AI
  • 4️⃣AI 基于真实数据作答，不再靠猜

效果是什么： AI 说的每句话都有原文依据。它会说"根据你的第 37 号内部文件……"，而不是"我认为……"


Larry 为产品参谋长🔍搭建的竞品情报 RAG，每天自动更新，实时检索，再也不用担心 AI 在用过时的信息。

"不上 RAG 的 AI 系统，就是把一个失忆症患者放在客服岗位上。他很聪明，但他不知道你公司发生了什么。"

— Larry
第四章

一个 AI 是助手，多个 AI 是组织。

单个 Agent 有局限：它的注意力有限，它的专业深度有限，它同时处理的任务数量有限。


就像一个全能员工，再厉害也做不到同时写代码、做分析、发内容。


多 Agent 团队的逻辑：分工。


Larry 的团队架构：

  • 👤总指挥（Larry）：定目标，做最终决策
  • 🔍产品参谋长：负责分析、洞察、竞品、用户
  • ⚙️技术专家：负责代码、部署、系统、自动化
  • ✍️笔杆子：负责内容、文案、发布、传播

三个 Agent 并行工作，各有专属 Skill，通过任务编排系统交接工作，Larry 只需要在关键节点做决策。


这不是科幻，这是工程问题。


每个 Agent 的边界要清晰，接口要定义好，结果要可验证。管 AI 团队，和管人类团队，本质没什么不同。就是更快，更便宜，不请假。

"AI 团队不是一个工具，是一套组织架构。架构设计得好，每多加一个 Agent，能力就指数级上升。"

— Larry
第五章

为什么同一个 AI，别人用得一塌糊涂，你用得行云流水？

不是你更聪明。是你的上下文设计得更好。


Context Engineering（上下文工程）是 Larry 认为 2024 年最被低估的 AI 技能。


它包含：

  • 👤系统角色定义：告诉 AI 它是谁、它的边界在哪
  • 📋任务框架设计：把复杂任务拆成 AI 能执行的步骤
  • 💉知识注入策略：什么时候注入什么背景信息
  • 📐输出格式约束：让 AI 输出你能直接用的结构
  • 🧠记忆管理机制：长对话中如何保持上下文不"忘事"

大多数人写 Prompt，只写了第一层：告诉 AI 要做什么。


Larry 写的是系统：五层上下文，每一层都有设计，每一层都有意图。结果是，他的 AI 团队第一次收到任务就能干对，而不是来回磨十几遍。

"Prompt 是一次性的对话，Context 是持续运行的系统。大多数人还在写对话，我在设计系统。"

— Larry
